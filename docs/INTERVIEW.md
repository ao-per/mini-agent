## 30 秒项目介绍

> 我实现了一个 Python 工具调用 Agent，没有直接依赖 Agent 框架，而是自己写了模型—工具—
> 模型的执行循环。它支持计算、时区时间查询和本地笔记搜索，并通过 Pydantic 做严格参数
> 校验。我重点补了工程化能力：模型异常映射、最大步骤、重复工具调用熔断、多轮会话、
> 离线假模型测试和真实模型最小评测集。这样既能验证控制流，又能衡量模型换版后的行为回归。

## 高频追问

### 为什么不直接使用 LangChain 或其他 Agent 框架？

这个项目的目标是理解并展示最小执行闭环，因此显式实现循环更容易观察消息协议、工具结果和
停止条件。生产项目如果需要大量集成、图编排或现成可观测性，我会重新评估框架，因为选择
框架应由复杂度和维护成本决定，而不是为了避免理解底层流程。

### 如何保证模型不会无限调用工具？

有两个边界：全局 `max_steps` 限制模型请求轮数；相同工具名和规范化参数组成 fingerprint，
超过阈值后不再执行，并向模型返回 `REPEATED_TOOL_CALL`。前者限制总成本，后者针对常见的
局部死循环。

### 为什么错误要返回给模型，而不是直接抛异常？

未知工具、参数错误和业务异常有时可以由模型修正，例如补充参数或改用其他工具，所以它们被
转换成结构化观察结果。模型服务不可用则结束本轮，并通过 `AgentResult` 把稳定错误码交给
调用方。这样区分了“模型可以恢复的工具错误”和“当前无法继续的基础设施错误”。

### 如何测试 Agent 而不调用真实模型？

Agent 依赖的是只有 `complete()` 方法的 `CompletionClient` 协议。测试注入
`ScriptedModelClient`，让它按顺序返回普通回答或工具调用，并检查 Agent 下一次发出的完整
消息。这样测试的是程序的编排逻辑，而不是模型能力。

### 单元测试已经通过，为什么还要评测？

单元测试只能证明“给定某个模型响应时程序行为正确”，不能证明真实模型会选对工具、填对参数
或给出合格答案。评测集固定用户任务，对真实模型的最终答案、工具轨迹和步骤数评分，用于发现
模型、提示词或 schema 变化造成的行为回归。

### 为什么第一版不用 LLM-as-a-judge？

当前案例大多有确定答案或明确工具要求，规则评分更透明、便宜且可复现。开放式任务增多后，
可以把人工抽检、规则评分和模型评分结合起来，同时保存 judge 版本和评分依据，避免把一个模型
的波动误认为被测 Agent 的变化。

### 这个项目最大的不足是什么？

目前没有持久化、流式输出、并行工具、权限审批和生产级可观测性，评测也以关键词为主。
如果继续做，我会优先增加 trace ID、延迟与 Token 指标、评测基线比较，然后为有副作用工具
增加权限分级和人工确认。

### 如果工具具有写操作，设计上要改什么？

需要把工具分为只读和有副作用两类；写操作执行前展示精确参数并要求确认；使用幂等键避免重复
提交；记录调用者、参数、结果和时间；为工具设置最小权限、超时与回滚策略。重复调用保护仍然
需要保留，但不能代替权限控制和幂等设计。

### 如何衡量 Agent 是否变好了？

至少同时看任务通过率、安全案例通过率、平均步骤、工具调用次数、延迟、Token 和错误分布。
每次模型、提示词或工具 schema 变化都与固定基线比较；对失败案例按“选错工具、参数错误、
答案错误、循环、服务错误”分类，避免只看一个总分。

## STAR 项目故事

### Situation

最初版本只能把用户问题发给模型，工具调用依赖模型自由输出，缺少异常边界，也无法稳定验证
修改有没有破坏行为。

### Task

目标是把它整理成一个可解释、可测试、可评测的最小 Agent，而不是继续堆叠功能。

### Action

拆分模型客户端、工具注册表、对话状态和 CLI；使用 Pydantic 校验工具输入；统一模型与工具
错误；增加最大步骤和重复调用保护；通过协议与依赖注入建立脚本化假模型；最后建立固定行为
评测集和 CI。

### Result

项目形成了从开发、离线测试到真实模型评测的完整闭环。目前有 3 个工具、10 条行为案例和
21 项离线测试，核心边界都有可复现验证。真实模型通过率应以生成的评测报告为准。
